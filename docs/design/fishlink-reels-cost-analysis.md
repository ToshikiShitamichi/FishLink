# FISHLINK 動画・リール機能 ランニングコスト試算（#199–202）

版: 2026-07-09 ／ 為替 1ドル=150円 ／ 現行スタック: Firebase（Storage＝Google Cloud Storage・Firestore・Cloud Functions・Hosting）

---

## 1. 結論（先に）

- **節約構成（クライアント圧縮＋タップ再生＋端末キャッシュ）なら、PoC〜本格初期（A〜C）はほぼ無料、拡大期（D＝農家50/レストラン200）でも月 ~$18（¥2,700）** で収まる。
- 増えるのは**ほぼ全部「動画のダウンロード帯域（egress）」**。保存容量・Firestore・Functions は全段階でほぼ $0。
- **リッチ構成（高画質＋フィード自動再生＋先読み）にすると拡大期で月 ~$135（¥20,000）前後**まで上がる。桁を決めるのは画質×自動再生×弱電波の再取得。→ **節約構成前提なら go に十分**。

## 2. 前提の整理

### 確定値
- 段階規模 A〜D（下記 §3）。
- 保持本数 **N=出品ごと最大10本**（`reels-spec.md` §6 確定・→5に下げ可）。差し替えは物理削除せずアーカイブ、N超過分の最古のみ削除。
- クリップ長 **5〜15秒**（仕様 §5・最大~30秒許容）。
- **アップロード帯域は無料**（GCS は保存＋ダウンロードのみ課金・ingress は非課金）。
- サムネ＝**出品の必須写真を流用**（動画サムネ用の追加 egress なし・仕様 §8）。
- 為替 **1ドル=150円**。

## 7. 重要な観察・注意点

- **主コスト＝ダウンロード帯域（egress）**。保存・Firestore・Functions は全段階でほぼ $0＝コストは「何本溜まるか」でなく「何回・どの画質で再生されるか」で決まる。
- **弱電波の効き**：`<video>` は Range/部分読み込みで再生し、再生反復＋自動再生の先読み＋切断→再取得で**実効 egress が1.8〜2.5倍**に膨らむ（リッチDが跳ねる主因）。かつ無料枠は「**1GB/日**」＝月平均で足りても繁忙日に日次超過し得る。
- **削減策（効く順）**：
  1. クライアント720p/低ビットレート圧縮（2〜6MB／元8〜30MB）。
  2. **サムネ（＝出品写真）先行表示・タップ再生**（カルーセルで自動再生しない）＝視聴本数を3〜5倍削減。
  3. 自動再生は**先読み1件のみ**・高速スクロールで中断。
  4. 保持 N 上限で保存を有界化（保存費は元々誤差だが青天井を防ぐ）。
  5. **端末Blobキャッシュ**で再生反復の再取得を $0 に（※現行SWは206/Range非キャッシュ＝要対処）。
  6. 定着後に**安価CDN前段**（bunny.net Asia ~$0.03/GB 等）で egress 単価を約1/4に。
- ⚠ **サーバ変換（Transcoder/Functions）に寄せると egress を上回る**＝480p圧縮は**クライアント側**に留める。

## 8. クライアント説明シナリオ

> PoC〜本格初期（農家5〜20・レストラン10〜50）は、動画を入れてもクラウド費用はほぼ増えません（**月0〜¥300程度**）。拡大期（農家50・レストラン200）でも、圧縮とタップ再生を効かせた「節約構成」なら**月¥3,000以内**に収まります。動画を高画質＋フィード自動再生の「リッチ構成」にすると拡大期で**月¥2万前後**まで上がり、増えるのはほぼ全部『再生のダウンロード帯域』です。**まずは節約構成で出し、視聴が伸びて弱回線のバッファリングが問題になったら安価な動画CDN（bunny.net 等）に載せ替える**——この順なら費用もUXも破綻しません。

## 10. 付録：大手（Instagram / Facebook 等）はどう配信しているか／PWA と ネイティブの違い

> 追記 2026-07-09（下道さん確認事項）。内部の正確な実装は非公開のため、**Meta Engineering ブログ等の公開情報＋業界標準（HLS/DASH・ABR・CDN）からの合理的な推測**。

### 10-1. 結論
大手は「バケットから MP4 を1本そのまま配る」方式では**ない**。中核は5点：**①アップロード時にサーバで多画質へ変換 ②ABR（適応ビットレート）で回線に合わせ画質を自動切替 ③自前/専用CDNでエッジ配信 ④次の動画を先読み ⑤新コーデック（AV1等）で圧縮**。我々のMVP（progressive MP4・タップ再生・クライアント圧縮）は、その**意図的に簡略化した部分集合**。

### 10-2. 大手の手法（公開情報ベース）
1. **サーバ変換で多ビットレート化**：Meta/Instagram は**全動画に ABR エンコードを生成**し、視聴予測が高い動画ほど高度エンコード（convex-hull 最適化）を優先（ML の benefit-cost で振り分け）。Instagram は基本ABRエンコードの計算コストを94%削減して適用範囲を広げた。
2. **ABR（HLS/DASH）**：動画を**2〜6秒のセグメントに分割**し 240p〜1080p の複数画質を用意、プレーヤーが実測帯域で**チャンク単位に画質を切替**＝弱回線は自動で低画質に落ちてバッファしない。← **我々の progressive MP4 との最大の差**。
3. **専用CDN/エッジ配信**：バケット直 egress でなく、ISP内に置いた自前アプライアンス（Meta の FNA 等）やエッジ POP から配信。だから彼らの egress 単価は我々の $0.12/GB とは別世界。
4. **先読み（prefetch）**：現在の動画＋レコメンドの**次の数本**を先読み＝スクロールで即再生（起動遅延 ~0）。帯域と視聴傾向で先読み量を動的調整（データ無駄を37〜52%削減する研究もある）。
5. **新コーデック**：AV1 / VP9 / H.265 で同画質を30〜50%小さく＝巨大スケールでの帯域削減。**CPU（エンコード）を使って帯域を買う**発想。

### 10-3. 大手 vs FISHLINK MVP

| 論点 | 大手（Meta/TikTok・ネイティブ） | FISHLINK MVP（PWA） |
|---|---|---|
| 変換 | サーバで多画質・多コーデック（AV1含む） | クライアント圧縮のみ・変換なし（Transcoderコスト回避） |
| 配信形式 | **ABR（HLS/DASH・セグメント）** | **progressive MP4 1ファイル** |
| 画質切替 | 回線に応じ自動 | 固定（480p等・保存前に決める） |
| CDN | 自前/専用CDN・ISP内キャッシュ | Firebase 直 egress（将来 bunny/Cloudflare 前段） |
| 先読み | 次の数本を予測先読み | **タップ再生（自動再生・先読みしない）** |
| プレーヤー | ネイティブ ExoPlayer/AVPlayer | `<video>`（＋将来 hls.js） |
| 狙い | 数億人・QoE最大化 | 数百人・コスト最小＆確実に動く |

→ 我々の割り切りは**規模と PWA/コスト制約に対して妥当**。**bunny.net / Cloudflare Stream / Mux は「この大手パイプライン（変換＋ABR＋CDN）をレンタル」**＝小チームでも②③⑤を数行で得られる＝§3・§7 の「定着後に載せ替え」推奨と直結。

### 10-4. PWA と ネイティブの違い（質問の核）
ここは**本当に差がある**。動画APIの「深さ」が違う。

- **ネイティブ（Meta 公式アプリはこちら）**：ExoPlayer(Android)/AVPlayer(iOS)＝ABR・バッファ制御・ハードウェアデコード・ディスクへの先読みキャッシュ・**バックグラウンド先読み**を細かく制御。TikTok級の「次を数本先読み」の滑らかさはネイティブだから出せる。
- **PWA/Web の制約**：
  - 🔴 **iOS が最大の壁**：iPhone Safari は長らく **MSE 非対応**＝hls.js 等で ABR を自前制御できず、**ネイティブHLS（`<video src=.m3u8>`）に頼るしかなかった**。2023/10 の **Safari 17.1 でようやく Managed Media Source（MMS）が iPhone に来た**が、**「AirPlayソース併記 or remote playback 無効化」が発動条件**という癖付き＝完全な自由度はない。
  - autoplay は **muted＋playsinline 必須**（iOS）。バックグラウンド先読みは実質不可。
  - キャッシュは HTTP cache / Service Worker / IndexedDB＝粒度が粗く、**SW は 206/Range を素直にキャッシュできない**（§5・§7 で挙げた落とし穴）。IndexedDB はクォータ制限＆自動退避あり。
  - 低レベルのコーデック/バッファ制御が無い＝ExoPlayer 相当の最適化はできない。
- ⚠ **重要**：Instagram を**「ブラウザで」開いた版（instagram.com）も、ネイティブアプリより制約された別実装**。web版は MSE/DASH＋自前プレーヤーで動くが、先読み・QoE はネイティブより控えめ＝**「大手ですら Web では同じ体験を出していない」**。だから我々が PWA で progressive MP4＋タップ再生に割り切るのは、Web の現実に沿った妥当な選択。

### 10-5. 我々への含意
- MVP は **「progressive MP4 ＋ タップ再生 ＋ クライアント圧縮」で正しい**（Web/PWA・コスト・低スペック端末の現実）。
- 「大手方式」に近づけたくなったら、自作でなく **bunny.net Stream / Cloudflare Stream / Mux**（変換＋ABR HLS＋CDN をまるごと提供）＝小チームで②③⑤を得る。iOS はネイティブHLS再生で問題なし。
- **ネイティブアプリ化（将来）**すれば ExoPlayer/AVPlayer で先読み・キャッシュが自由になり TikTok 的滑らかさに最も近づく＝ただし PWA 戦略を捨てる判断とセット。
