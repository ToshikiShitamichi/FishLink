# FISHLINK 公開質問（Q&A）画面 実装仕様

実装の正本は **`fishlink-qa-mockup.html`**（構造・レイアウト・配置はそこを参照）。
本書はHTMLから読み取れない情報＝**データモデル・権限・通知・中抜き対策・モデレーション・状態・遷移・制約**を補う。

> 既存実装は構成（A案）が良好（1質問＝時系列1段スレッド・往復フラット・もっと見る折りたたみ・取り消しは自分のみ・公開注意・空状態）。差分は §9。

### 実装方針の優先順位（他specと同じ）
- 構成・データ・権限・通知・状態・遷移 → 本書とモック通り。
- 見た目（色・タイポ・アイコン）→ 既存DSに準拠。**送信ボタン（質問する／返信する）は青 `#0B5FB0`**（赤はdanger専用）。

---

## 1. 概要

商品（出品）に対する**公開Q&A**。レストランが農家に質問し、農家が返信する。**他のレストランも閲覧でき**、注文前の確認に使う。商品詳細の「質問をすべて見る（◯件）→」／「この魚について質問する →」から遷移。

- **構成＝A案**：1質問＝1スレッド。スレッド内は**質問者と農家の往復を時系列フラット（1段インデント）**で表示（深いネスト無し）。
- リピート単位は魚種だが、Q&Aは**出品（商品）単位**。

---

## 2. 画面構成

ヘッダー（戻る＋ロゴ＋言語）→ タイトル「質問」→ **商品コンテキスト箱**（タップで商品詳細へ）→ 公開注意 → スレッド一覧 → 固定フッター（質問/返信の入力＋送信ボタン）。

- **商品コンテキスト箱**：魚名（サイズ）＋農家名＋**価格**。**タップで商品詳細へ遷移**。
- **公開注意**：「質問は他のレストランにも公開されます。注文前の確認に利用してください。」
- **スレッド**：質問（質問者・相対時刻・質問バッジ・本文）＋返信（農家/質問者・返信バッジ・本文）を時系列フラットに。長いスレッドは **「もっと見る（あと◯件の返信）」** で折りたたみ（質問＋最新返信のみ表示）。
- **スレッド並び順＝新しい質問が上**。
- **空状態**：スレッド0件のとき「**まだ質問はありません**」＋「最初の質問をどうぞ」。商品箱・入力欄は残す。

---

## 3. データモデル

```js
qaThread = {
  id, listingId,
  question: { authorId, authorName, text, createdAt },
  replies: [ { authorId, authorName, authorRole /* "asker" | "farmer" */, text, createdAt } ],
  // 表示は question + replies を createdAt 昇順でフラットに並べる（1段）
}
```
- ヘッダー価格は **listing の買い手向け表示価格（手数料込み・内臓処理込み＝全画面で単一の正）**を参照（§9-価格）。

---

## 4. 権限（返信・削除・新規）

- **新規質問（スレッド作成）**：ログイン中のレストランは誰でも可（注文前でも可）。下部「質問を入力／質問する」。
- **スレッドへの返信（返信する）**：**そのスレッドの質問者本人**と**農家**の2者のみ。
  - **農家** → 全スレッドに返信可（「返信する」表示）。
  - **質問者本人** → 自分が立てたスレッドのみ「返信する」表示。
  - **その他のレストラン** → 「返信する」**非表示**（閲覧のみ。疑問は新スレッドで）。モックでは「閲覧のみ（自分の質問ではないため返信できません）」の注記。
- **取り消し（削除）**：閲覧者**自身が書いた投稿（質問 or 返信）にのみ**「取り消し」表示。
  - **質問（スレッド起点）を取り消すと、スレッドごと消える**（農家の返信も含む。起点が消えると返信だけ残っても文脈が無いため）。返信単体の取り消しはその返信のみ消える。
- スレッド内の往復は**質問者と農家の2者間のみ**（3者目は入れない）。

---

## 5. 通知

- **農家へ**：自分の出品に**新しい質問**が来たら通知（🔔＋プッシュ）。**未回答の質問は農家の「やることリスト（✓✓）」に積む**（放置＝失注の防止）。
- **質問者へ**：自分の質問・スレッドに**農家が返信**したら通知。
- スレッド内の新規投稿は**相手（2者のうちもう一方）にのみ**通知。他レストラン（閲覧者）には通知しない。
- 連投の集約はMVPでは都度通知でも可（実装判断）。

---

## 6. 中抜き（プラットフォーム外取引）対策＝B案

- 公開Q&A・配送管理チャットは農家↔レストランが直接やり取りする場 → **連絡先交換→直接取引で手数料収入が抜ける**リスク。
- **B案＝連絡先の自動マスク＋投稿時警告**：
  - 質問/返信の本文から**電話番号・SNSアカウント・URL**を検知し、**伏字（［連絡先は非表示］）にして投稿**。
  - 投稿時に警告「**電話番号・SNS・URLなどの連絡先は投稿できません。やり取りはアプリ内でお願いします。**」（アンバー）。
  - **適用範囲＝公開Q&A＋配送管理チャット の両方**。
- C案（運営監視・違反ペナルティ・アカウント制限）は運用が回ってから。
- ※完全には防げない（対面・口頭は無理）が、公然の場での交換を塞ぐ抑止。本質的防御は「プラットフォームにいる価値」（決済・掛け払い・配送手配・トラブル保証・履歴/再注文）＝事業設計側の別論点。

---

## 7. モデレーション（運営削除）

- **マスク発動投稿を、管理画面「トラブル報告」に新カテゴリ「公開質問（要確認）」として自動表示**（既存のトラブル報告の枠組みに乗せる。新画面は作らない）。
  - カード＝検知日時／関連商品／投稿者／**マスク済み本文**＋アクション。
  - アクション：投稿者とチャット／詳細を見る／**「🗑 投稿を非表示にする」**／対応済にする。
- **「投稿を非表示にする」＝ソフト削除**（公開から消すが**DBには残す**。「運営により削除」記録）。物理削除しない（証拠・履歴保持）。
- 権限は**adminロールのみ**。**ユーザーからの通報導線（報告ボタン）はMVPでは付けない**（B2Bで荒れにくい・件数少・運営目視で足りる。増えたら追加）。
- in-context（公開Q&Aページ上でadmin限定の非表示ボタン）は後で追加可。
