## 1. 概要

農家が**受注に承認/辞退で応答**し、承認後は**配達を管理**する画面群。買い手の発注（カート→注文確定）で**農家ごとに独立した注文**が作られ、ここに届く。

- 在庫の**最終真実は農家の承認**（カートは軽量ガードのみ＝cart spec §5）。
- 注文は**農家ごと独立**＝承認/辞退も個別（片方成立・片方辞退できる）。

## 3. データモデル

```js
farmerOrder = {
  id,                 // "FL-260601-003"
  fishName, sizeHeadPerKg,
  unitPriceKhr,       // 魚単価（農家の基本単価。例 7,000）
  gutFeeKhr,          // 内臓処理の農家取り分（例 500）
  gutIncluded,        // この注文が内臓処理ありか
  qtyKg,
  deliveryDate, deliverySlot,
  restaurant: { name, ratingPositivePct, ratingCount, province, district, distanceKm, mapUrl },
  approveDeadline,    // 承認期限（注文時刻 + leadHours。既定1h・運営パラメータ）
  status,             // 下記ライフサイクル
  declineReason,      // 辞退時：enum＋自由記述
  deliveryFeeKhr,     // 送料（農家へ全額パススルー）
}

status = "awaiting"     // 承認待ち
       | "preparing"    // 準備中（承認後）
       | "delivering"   // 配送中
       | "arrived"      // 到着（買い手に受取依頼）
       | "done"         // 完了（配送完了）→ レビュー解禁
       | "declined"     // 辞退（手動 or 期限切れ自動）
       | "awaiting_payout" // 送金待ち（完了後の精算）

declineReason.code = "out_of_stock" | "size_mismatch" | "time_unavailable" | "delivery_hard" | "other"
```

## 4. 金額計算（農家視点）★

- **魚代** = `(unitPriceKhr + (gutIncluded ? gutFeeKhr : 0)) × qtyKg`。
  - 例：あり 7,000＋500＝7,500 → ×5 ＝ **37,500**。なしなら 7,000×5＝35,000。
  - **②詳細では魚単価と内臓処理を分解表示**（「魚単価 7,000／内臓処理 +500」）＝農家は実際に内臓処理をやるので対価と分かる＋出品フォームと整合。
- **手数料** = `魚代 × serviceRatePct`（通常率＝例5%）。表示は **「手数料（5%）−1,875」**（5%の実額を出す）。
- **キャンペーン割引** = `魚代 × campaignDiscountPct`（例2.5%）を **戻す（+表示）**。表示「キャンペーン割引 +938」。
  - **手数料は通常率の満額を出し、キャンペーンで戻す**形（ラベル「5%」と額のズレを作らない）。
- **送料** = `deliveryFeeKhr`。**農家へ全額パススルー**（プラットフォームは送料から手数料を取らない＝手数料は魚代のみ）。表示「送料 +12,400（レストラン負担・農家受取）」。
- **受取予定額** = `魚代 − 手数料 + キャンペーン割引 + 送料`。
  - 例：37,500 − 1,875 + 938 + 12,400 ＝ **48,963**。
- 価格モデルは買い手側と同根（買い手価格＝農家販売価格×(1+手数料率)）。**全画面で価格は単一の正**。

## 5. ステータス・ライフサイクルと買い手側マッピング

農家は配達を**細かい操作**で進め、買い手のHome 5段ステッパー（注文済/農家確認中/出荷/配達中/完了）に**マッピング**して反映する。

| 農家の状態・操作 | 買い手の表示（Home 5段） |
|---|---|
| 承認待ち | 農家確認中 |
| 承認 → 準備中／準備開始 | 出荷（準備） |
| 配送開始 | 配達中 |
| 少し遅れそうです | 配達中（＋「遅延の可能性」表示） |
| **到着** | **配達中のまま**＋ステッパー下のメッセージ行に「📍 到着しました・お受け取りください」＋**プッシュ通知**（ステッパーに到着ノードは作らない／"まもなく"は使わない＝到着＝もう着いた） |
| 配送完了 | **完了** ← レビュー解禁 |
| 辞退（手動/自動） | 辞退（赤枠・「他の農家を探す」） |

- **注文ステータスは単一の真実**を両アプリで共有し、表示文言だけ役割別にする。

## 6. 承認 / 辞退フロー

- **承認/辞退の二択**（MVP）。**部分承認（数量を減らして承認）はしない**＝在庫不足は辞退。
  - ※**数量の部分承認は次フェーズ最優先**。データは「**確定数量（≠注文数量でありうる）**」を持てる形に布石（後付けしやすく）。サイズ・時間の代替提案はさらに後 or やらない。
- **承認確認モーダル（自前）**：送信前に注文サマリ（魚/数量(内臓処理あり/なし)/納品日時/レストラン/受取予定額）＋「レストランに通知が届きます」を出す。標準ブラウザダイアログは使わない（カート削除確認も含め共通改善）。
- **辞退**：理由を**構造化選択**（在庫が足りない／希望サイズに合わない／納品時間に対応できない／配送が難しい／その他）＋詳細(任意)。買い手は「他の農家を探す」へ。理由はデータ化（分析・農家指標に活用余地）。
- **承認期限＝1時間（運営パラメータ）**。**10分前にリマインドpush**（実装済）。**期限切れ → 自動辞退**（→買い手「他の農家を探す」／農家へ「期限切れで自動辞退」通知）。自動辞退が多発するなら期限を延ばす（数字変更のみ）。**無応答ペナルティ（評価反映）は後フェーズ**。

---

## 7. 配送管理（⑤）

- **チャット**：農家↔レストラン。**音声メッセージ対応**（低リテラシー層に有効）＋テキスト＋写真。
- **配送ステータス送信ボタン**：準備開始 / 配送開始 / 少し遅れそうです / 到着 → §5 のマッピングで買い手に反映。
- **「配送完了」ボタン**：農家がこれを押すと **完了** に遷移 → 買い手「完了」＋**レビュー解禁**。
- 完了後は **送金待ち**（精算）へ。※送金待ち/精算フローは別途レビュー。

## 8. 表示ルール

- **評価＝`★{好評価率}%（件数）`**（例「★93%（40件）」）。**小数（93.0%）は使わない**＝#1の%統一を農家側にも適用。
- **魚単価は分解表示**（魚単価／内臓処理 +500）＝§4。
- **CTAは青塗り**（送信する／配送完了／承認する／チャット送信）。**問題を報告＝赤枠（danger）維持**。オレンジは使わない（キャンペーン色と混線）。
- **受取予定額（合計の金額）は青 `#0B5FB0`**（金額の主役。赤/オレンジにしない＝赤はdanger専用。買い手のカート合計＝青と同じ思想。2026-06-03 実装で赤/オレンジだった→要修正）。
- **レビュー送信は「完了」後のみ解禁**（準備中・承認済み段階では出さない）。
- 内臓処理の表記は「（内臓処理あり）/（内臓処理なし）」とフル表記（"(あり)"だけにしない）。
- **キャンペーン割引（「◯◯ お得」）は amber（🏷＋オレンジ文字）に統一**（買い手のカート/注文確認の割引と同じ。緑にしない＝緑はGAqP/成功/送料無料専用。2026-06-03 実装で緑だった→要修正）。
- **納品日は曜日付き表記「6/5（◯）」**（ISO「6/5」だけにしない。カート/注文確認/Homeと統一）。農家側の注文確認・承認モーダル・配送管理に適用。
