## 1. 概要

農家が**受注に承認/辞退で応答**し、承認後は**配達を管理**する画面群。買い手の発注（カート→注文確定）で**農家ごとに独立した注文**が作られ、ここに届く。

- 在庫の**最終真実は農家の承認**（カートは軽量ガードのみ＝cart spec §5）。
- 注文は**農家ごと独立**＝承認/辞退も個別（片方成立・片方辞退できる）。

## 1.5 【案B 前払い対応＝2026-06-26】

> お金のロジックの正本は `fishlink-payment-spec.md`（案B v2.0）。本specは農家画面のUI/挙動に限定。mockは `fishlink-farmer-order-mockup.html`（8画面）が正本。

**前払い済みの可視化（案B最大の農家メリット）** — レストランは注文時に前払い済み＝「承認して届ければ確実に受け取れる」。
- ① 承認待ちカードに**「✓ 支払い済み（前払い）」**（緑・FL番号の下）。
- ② 承認待ち詳細の金額ブロック上に**緑の支払い済み帯**「✓ 支払い済み（前払い）／レストランのお支払いは完了しています。受取予定額は配送完了後に送金されます」。
- ③ 承認確認モーダルの注記に「**レストランは支払い済みです。**承認すると…」。

**受取時払いの名残を修正**
- ① 到着カード「お受け取り・**お支払い**をお待ちください」→「**お受け取りをお待ちください**」（受取時に金は発生しない）。
- 辞退確認モーダル「この注文は**他の農家に回されます**」（誤）→「辞退すると、レストランに通知され、**お支払いは返金されます**」（買い手は即ウォレット返金＝§4.5・自動ルーティングではない）。

**配送完了後・送金保留（payment-spec §4.6・§8）**
- ⑤' **配送完了後**：配送管理チャットに緑の完了システムメッセージ「✓ 配送完了。問題がなければ数時間後に取引完了・送金確定します」＋フッター完了バナー。
- ⑥ 売上・取引：問題報告で送金保留の注文は**「確認中」バッジ**（受取予定額（保留中）＋⚠問題が報告されています）＋サマリを「送金待ち（本日送金予定）」と「**確認中（問題報告あり・保留）**」に**別枠分離**。タップで精算の送金状況（確認中）へ。

**承認後に農家がキャンセルしたい時** — 専用ボタンは作らず**運営チャット経由**（買い手はウォレット返金・payment-spec §9）。

**不変** — 受取額・手数料・送金タイミング（配送完了後）は変わらない（早まったのは買い手が払う瞬間だけ）。データモデルの `awaiting_payout`（送金待ち）は維持＋`payout_holding`（確認中）を追加（payment-spec §3）。

## 2B. 注文確認リスト（A案・優先フィード型）

**画面の正体＝「やることキュー」**。最優先制約＝**承認待ちを見逃させない＝最上部固定**（低リテラシー農家・期限切れ自動辞退＝失注を防ぐ）。タブ型は却下（裏に隠れて見逃す）。

**構成（上から）**：
1. **対応が必要（承認待ち）**＝赤見出し。**要約カード**（サムネ／📅日時（曜日付き）／魚名(size)・数量／レストラン／FL番号／⏰承認期限カウントダウン（赤）／**「対応する」→ 承認待ち詳細②**）。承認期限の近い順。urgent枠（赤系ハイライト）。
2. **進行中**＝承認済み（準備中）・配送中・**到着**。カード＝状態バッジ＋「**配送管理へ**」。
   - **状態バッジ**：準備中＝グレー／配送中＝青／**到着＝ティール「📍到着」**。
   - **到着**＝農家は配達済みで**買い手の受け取り（＝KHQR支払い）待ち**。カードに状態文「**📦 お受け取り・お支払いをお待ちください**」（ボールは買い手側＝農家は待つだけ／“届けたのに未払い”の気づきにもなる）。アクションは「配送管理へ」のまま。
3. **レビュー**＝「**⭐ レビュー待ち ◯件 ›**」の1行（見出し「レビュー」）。**取引完了かつ農家が未レビュー・直近2週間**の取引がある時だけ表示（0件は見出しごと非表示）。タップ → **レビュー待ち一覧（子画面 ①'）**＝該当取引を軽いカードで並べ各カードに「⭐ レビューを送信」→ レビュー入力画面（review-spec §7.1）。**A案で完了カードを撤去しレビュー入口が送金状況の奥に潜った問題の解消**＝毎日見る注文確認に主導線を戻す（送金状況の「レビューを送信」＝文脈導線は残す／pushと合わせ3層）。※2週間超は入口が消えるだけ＝送金状況からは引き続きレビュー可（期限ではない）。mock＝①'。
4. **履歴**＝「**🧾 取引履歴を見る（送金待ち N・完了 N・辞退 N）›**」の**1行リンク**（→⑥売上・取引）。アコーディオン3本は廃止。

**空状態**：①0件→「対応が必要」見出しごと非表示（進行中が最上部に）／②0件→「進行中」非表示／**レビュー待ち0件→「レビュー」見出しごと非表示**／**対応中ゼロ（①②とも0）→「✓ 対応が必要な注文はありません」**（赤くしない＝片付いてる合図）＋レビュー待ち（あれば）・取引履歴行は残す／**完全新規（注文ゼロ＋履歴ゼロ）→「まだ注文がありません」＋出品導線**。

**カード本体タップ＝そのカードの主アクション**で統一：承認待ち→対応する（②）／進行中→配送管理（⑤）／売上・取引の履歴カード→送金状況（§7B）。

## 3. データモデル

```js
farmerOrder = {
  id,                 // "FL-260601-003"
  fishName, sizeHeadPerKg,
  unitPriceKhr,       // 魚単価（農家の基本単価。例 7,000）
  gutFeeKhr,          // 内臓処理の農家取り分（例 500）
  gutIncluded,        // この注文が内臓処理ありか
  qtyKg,
  deliveryDate, deliverySlot,
  restaurant: { name, ratingPositivePct, ratingCount, province, district, distanceKm, mapUrl },
  approveDeadline,    // 承認期限（注文時刻 + leadHours。既定1h・運営パラメータ）
  status,             // 下記ライフサイクル
  declineReason,      // 辞退時：enum＋自由記述
  deliveryFeeKhr,     // 送料（農家へ全額パススルー）
}

status = "awaiting"     // 承認待ち
       | "preparing"    // 準備中（承認後）
       | "delivering"   // 配送中
       | "arrived"      // 到着（買い手に受取依頼）
       | "done"         // 完了（配送完了）→ レビュー解禁
       | "declined"     // 辞退（手動 or 期限切れ自動）
       | "awaiting_payout" // 送金待ち（完了後の精算）

declineReason.code = "out_of_stock" | "size_mismatch" | "time_unavailable" | "delivery_hard" | "other"
```

## 4. 金額計算（農家視点）★

- **魚代** = `(unitPriceKhr + (gutIncluded ? gutFeeKhr : 0)) × qtyKg`。
  - 例：あり 7,000＋500＝7,500 → ×5 ＝ **37,500**。なしなら 7,000×5＝35,000。
  - **②詳細では魚単価と内臓処理を分解表示**（「魚単価 7,000／内臓処理 +500」）＝農家は実際に内臓処理をやるので対価と分かる＋出品フォームと整合。
- **手数料** = `魚代 × serviceRatePct`（通常率＝例5%）。表示は **「手数料（5%）−1,875」**（5%の実額を出す）。
- **キャンペーン割引** = `魚代 × campaignDiscountPct`（例2.5%）を **戻す（+表示）**。表示「キャンペーン割引 +938」。
  - **手数料は通常率の満額を出し、キャンペーンで戻す**形（ラベル「5%」と額のズレを作らない）。
- **送料** = `deliveryFeeKhr`。**農家へ全額パススルー**（プラットフォームは送料から手数料を取らない＝手数料は魚代のみ）。表示「送料 +12,400（レストラン負担・農家受取）」。
- **受取予定額** = `魚代 − 手数料 + キャンペーン割引 + 送料`。
  - 例：37,500 − 1,875 + 938 + 12,400 ＝ **48,963**。
- 価格モデルは買い手側と同根（買い手価格＝農家販売価格×(1+手数料率)）。**全画面で価格は単一の正**。

## 5. ステータス・ライフサイクルと買い手側マッピング

農家は配達を**細かい操作**で進め、買い手のHome 5段ステッパー（注文済/農家確認中/出荷/配達中/完了）に**マッピング**して反映する。

| 農家の状態・操作 | 買い手の表示（Home 5段） |
|---|---|
| 承認待ち | 農家確認中 |
| 承認 → 準備中／準備開始 | 出荷（準備） |
| 配送開始 | 配達中 |
| 少し遅れそうです | 配達中（＋「遅延の可能性」表示） |
| **到着** | **配達中のまま**＋ステッパー下のメッセージ行に「📍 到着しました・お受け取りください」＋**プッシュ通知**（ステッパーに到着ノードは作らない／"まもなく"は使わない＝到着＝もう着いた） |

| 辞退（手動/自動） | 辞退（赤枠・「他の農家を探す」） |

## 6. 承認 / 辞退フロー

- **承認/辞退の二択**（MVP）。**部分承認（数量を減らして承認）はしない**＝在庫不足は辞退。
  - ※**数量の部分承認は次フェーズ最優先**。データは「**確定数量（≠注文数量でありうる）**」を持てる形に布石（後付けしやすく）。サイズ・時間の代替提案はさらに後 or やらない。
- **承認確認モーダル（自前）**：送信前に注文サマリ（魚/数量(内臓処理あり/なし)/納品日時/レストラン/受取予定額）＋「レストランに通知が届きます」を出す。標準ブラウザダイアログは使わない（カート削除確認も含め共通改善）。
- **辞退**：理由を**構造化選択**（在庫が足りない／希望サイズに合わない／納品時間に対応できない／配送が難しい／その他）＋詳細(任意)。買い手は「他の農家を探す」へ。理由はデータ化（分析・農家指標に活用余地）。
- **承認期限＝1時間（運営パラメータ）**。**10分前にリマインドpush**（実装済）。**期限切れ → 自動辞退**（→買い手「他の農家を探す」／農家へ「期限切れで自動辞退」通知）。自動辞退が多発するなら期限を延ばす（数字変更のみ）。**無応答ペナルティ（評価反映）は後フェーズ**。
- **買い手のキャンセル受領（承認前のみ）**：買い手は**農家の承認前なら注文をキャンセルできる**（承認後は不可）。キャンセルされたら**農家へプッシュ通知＋承認待ちリストからそのカードを消す（承認期限停止）**。**競合ロック＝同一注文に対し承認とキャンセルは二重成立しない**（先に押した方が成立。買い手キャンセルが先→農家の「承認する」は弾く「この注文はキャンセルされました」）。記録＝取引履歴に**「キャンセル」**（辞退＝農家都合とは別ラベル・⑦の成立しなかった注文に灰バッジで入る）。

## 7. 配送管理（⑤）

> **チャットの詳細仕様は `fishlink-chat-spec.md`（配送管理チャット）に分離**（両アプリ横断＝レストラン側と共通のため）。モックは `fishlink-chat-mockup.html`（左=レストラン/右=農家）。本節は農家側⑤の要点のみ。システム行のラインアイコン化・日付区切り・音声UI（▶＋波形＋秒数）・押下制御の見た目はチャットspec §4/§5・§11 を参照。

- **チャット**：農家↔レストラン。**音声メッセージ対応**（低リテラシー層に有効）＋テキスト＋写真。
  - **ライフサイクル**：**承認後（準備中〜）に開く**（承認待ち中はチャット無し＝事前の疑問はQ&A）。**配送完了後はユーザーにはチャットUIを閉じる**（完了後カードは「配送管理へ／配送状況を確認」→「送金状況/支払い状況を見る」に切替＝現状でOK）。**ただしチャット履歴はサーバーに保持（削除しない）し、運営が問題報告の対応時に閲覧できること（必須）**＝紛争の証拠（配達写真/到着時刻/約束事）。ユーザー向けの完了後 read-only 表示は後フェーズ（MVP省略）。
  - **通知＝新着が来たら相手にプッシュ通知**（農家が送ったらレストランへ／レストランが送ったら農家へ。メッセージ・配送ステータス更新・音声・画像すべて同じ＝来たら通知）。**集約（連投まとめ）はしない**（MVPは都度通知）。やることリスト（✓✓）連動は別途整理（後回し）。
  - **連絡先マスク（中抜き対策）をチャットにも適用**：電話番号・SNS・URLを検知→伏字＋警告（公開Q&Aと共通＝Q&A spec §6）。
  - **📞通話ボタン（ヘッダー）＝原則「実番号を出さない」**。理想＝**マスク通話（プロキシ・双方に実番号を見せずアプリ経由）**（中抜き防止＝FISHLINKの核＝仲買人排除と整合。※CAA価格表の番号は仲買人のもので農家番号は非公開）。マスク通話の基盤が用意できない場合のMVP割り切り＝**📞は「注文がアクティブな間（承認後〜完了まで）有効・完了後は消す」**（"配達中"ステータス限定にしない＝手動ステータス未設定で電話できない問題を回避・露出を進行中注文に限定）。**実番号を恒久的に表示/発信するのは不可**。後でマスク通話に昇格。
- **配送ステータス送信ボタン**：準備開始 / 配送開始 / 少し遅れそうです / 到着 → チャットにメッセージとして流れる＋**§5 のマッピングで買い手の5段ステッパーにも反映**（チャット文だけでなくステータスも進める）。
  - **押下制御＝緩い誘導（B案）**：進行順（準備開始→配送開始→到着）に沿って**次に押すべきボタンを強調**するが、**押下自体は自由**（ロックしない）。理由＝鮮魚配達は段取りが前後する（準備せず即配送等）ので厳密固定は硬すぎる。「**少し遅れそうです**」は進行と無関係に**いつでも押せる**割り込み。重複押しは許容（買い手には最新が反映）。

## 8. 表示ルール

- **評価＝`★{好評価率}%（件数）`**（例「★93%（40件）」）。**小数（93.0%）は使わない**＝#1の%統一を農家側にも適用。

- **受取予定額（合計の金額）は青 `#0B5FB0`**（金額の主役。赤/オレンジにしない＝赤はdanger専用。買い手のカート合計＝青と同じ思想。2026-06-03 実装で赤/オレンジだった→要修正）。

- **送金状況画面のレビュー送信＝副CTA（白地青枠）**・問題を報告＝赤（報告できる期間だけ）。レビューは**支払い確認後のみ解禁**。
