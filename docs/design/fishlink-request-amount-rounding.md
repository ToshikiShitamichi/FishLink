# 【実装依頼】KHR金額の丸め＝100リエル単位・切り捨て（2026-07-04）

## 🔴【2026-07-22 追記＝出品フォームの単価入力が100単位を強制していない】
出品フォーム「魚を出品する」の価格入力が、**案内文（「100リエル単位で入力してください」）だけで、入力を強制していません**（＝本依頼「単価は100刻み入力に」がまだ効いていない）。
- **実機で確認**：**魚単価に「7,550」が通る**／**内臓処理に「555」が通る** → 販売価格が **8,105 KHR/kg**（＝7,550+555・非100単位）になる。
- **なぜマズいか**：単価が100単位でないと **魚代＝単価×kg も100単位からズレる**（例 7,550×5=37,750）＝本依頼の「切り捨てるのは手数料・割引の行だけ・他は自然に揃う」前提が崩れる。土台の部分です。
- **修正＝魚単価・内臓処理の入力を100単位に強制**（両方とも）。**方向＝一番近い100にスナップ**（例 7,550→7,600／555→600）。※農家が自分で設定する売値なので、切り捨てでなく"最寄り100の四捨五入"が自然（ウォレットの切り捨てとは別＝あちらは「使う額」で残りが戻るため）。blur時にスナップ or 100刻みステッパーで。
- **併せて＝送料も100単位に（配送設定）**：送料は"計算値"（**10kmあたりのレート × 距離**）なので2箇所：
  - **① 実際の送料（買い手が払う・農家パススルーの額）＝100リエル切り捨て**（必須）。理由＝距離が0.1km刻み（例 24.8km）だと、レートが100単位でも計算結果が50単位になりうる（例 24.9km×500=12,450）。お金が動く額なので100単位に。切り捨て＝買い手有利／送料は農家パススルーなので買い手も農家も同じ丸め値（誰も損しない）。
  - **② 10kmあたりの送料（レート入力）＝100単位に強制**（魚単価と同じ・最寄り100スナップ）。※最終額は①で丸めるので副次的だが、レートも綺麗に。
  - ※配送可能距離・送料無料距離は「km（距離）」なので100単位の対象外。

## 🔴【2026-07-17 追記＝実機で「半分だけ実装」を発見・直してほしい】
実機のカートが**「単価の表示だけ丸めて、計算は丸める前の値」**になっています＝本依頼の「カートは単価を丸める方式で」がまだ効いていません。
- **実例（スポットパンガシウス）**：単価表示 **7,300** KHR/kg × 10kg なのに、小計が **73,500**（＝丸める前の7,350で計算している）。買い手が検算すると合わない（7,300×10=73,000のはず）。
- **修正＝表示している丸めた単価（7,300）で小計・合計も計算する**（＝本依頼の本文どおり。「表示だけ丸め」「合計だけ丸め」はどちらもNG）。
- 数量1の商品ではズレが見えないだけで、**数量2以上で必ず露見**します。
- ※魚代・合計・お支払い額など他の行は100単位で整合しており、残るのはこの1点です。
- 併せて軽微：確認画面のウォレット変更を開いた状態の「お支払い額 **41.000** KHR」＝カンマがピリオドになっている表記ゆれ（他は 41,000）。
- **併せて＝ウォレット利用の"手入力欄"も100リエル単位に制限**：KHQR支払い額＝`合計 − ウォレット利用`。合計もKHQR額も100単位でないと成立しないので、**ウォレット利用も100単位に揃える**（例：ウォレットに50,750と入れると 91,800−50,750＝41,050 で100単位から外れ、銀行で払えない）。100未満は**切り捨て**（残りはウォレットに残る＝買い手有利・全体ルールと同じ向き）。クイックボタン〔全額〕〔使わない(0)〕は元々100単位なので、**手入力欄だけ**直せば全経路（KHQR・送金・返金）が100単位に揃う。

下道さん

金額の桁について1つルールを決めました。**FISHLINKのKHR金額はすべて100リエル単位（100の倍数）に揃えます。** 正本＝`fishlink-payment-spec.md` §4.2。

## 背景（なぜ）
- カンボジアのKHRは**実務上100リエルが最小単位**（硬貨はほぼ不使用・100が最小紙幣・銀行送金も100単位強制）。
- いまアプリの金額（例 48,963 / 50,791 KHR）が100で割り切れないのは、**手数料5%やキャンペーン/クーポン割引（％計算）が端数を生む**ため。
- 実際にお金が動く3か所は100単位でないと成立しない：**①買い手のKHQR支払い ②農家への送金 ③ウォレット返金**。表示と実額がズレると農家の信頼を損なう。

## ルール（これを実装）
> **KHR金額は100リエル単位。端数を生む「手数料・割引（％計算）の行」で、100リエル未満を切り捨てる。端数差（最大99リエル）は運営が吸収。**

### 実装の肝
1. **単価（KHR/kg）と送料は100刻みの入力に**（＝魚代＝単価×整数kg は自動で100単位）。
2. **切り捨てるのは手数料・割引の計算結果だけ**：
   ```
   魚代 37,500（100単位）
   送料 12,400（100単位）
   手数料 5% → 1,875 → 【切り捨て 1,800】
   キャンペーン → 938 → 【切り捨て 900】
   ──────────────
   受取額 = 37,500 − 1,800 + 900 + 12,400 = 49,000 ✓100単位
   ```
3. **行ごとに丸める**（最後の合計だけ丸めない）＝買い手の支払額・農家の受取額・ウォレット返金・帳簿がすべて自動で一致。

### なぜ「切り捨て」か
**手数料が約束の率（標準5%）を絶対に超えないため。** 四捨五入だと 1,875→1,900 で実効5.07%になり得る＝「言った率より多く取る」構造を避ける。切り捨てなら常に5%以下。差額は運営が飲む（農家・買い手は常に損しない）。

## 買い手表示価格・カートも100リエル単位（重要）
農家が100刻みで入力しても、**買い手向けは ×（1＋手数料）で端数が復活**します。ここも100リエルに揃えます。

- **買い手のkg単価 ＝ 切り捨て（販売価格 ×（1＋手数料率）, 100）**。例：販売価格7,500 → 7,875 → **7,800 KHR/kg**。内臓処理の減額（外す）も 500×手数料率＝525 → **500**。
- **カートは「単価を丸める」方式で**（＝上の丸めた単価で計算）。**「合計だけ丸める」はNG**。
  - 実機で発覚：単価7,875（端数のまま）×2 ＝15,750 なのに合計だけ切り捨てて **15,700** と表示＝**単価×数量（15,750）≠合計（15,700）で計算が合わない**。買い手が「2個で7,875なのに、なぜ15,700？」と混乱。
  - 正：単価を7,800に丸める → 7,800×2＝15,600（単価×数量＝合計で一致）。行小計・魚代・キャンペーン割引（％も100切り捨て）・小計・合計すべて100単位で整合。
  - お手本＝`fishlink-cart-mockup.html`（①合計103,200／②102,900・全部100刻み＆単価×数量＝小計で整合）／`fishlink-cart-spec.md` §4。

## 確認してほしいこと
- **KHQR（買い手がQRで払う）が端数額を許すか**（＝50,791をそのまま請求できるか）。許す場合でも、**表示・送金・返金は100単位に統一**する方針。農家payoutは銀行送金なので100単位はほぼ必須。ABA/Bakong側の仕様を確認いただけると助かります。

## 補足
- **各モックの金額は、まだ端数のまま（48,963 等）です**。これはルール確定前の値で、**100単位への一括修正（sweep）は後追い**でやります。実装は上のルールが正、モックの端数値は参考にしないでください（正本＝payment-spec §4.2）。
