# FISHLINK 精算フロー（決済・送金）実装仕様

実装の正本は **`fishlink-payment-mockup.html`**（6画面・構造/レイアウト/配置はそこを参照）。
本書はHTMLから読み取れない情報＝**状態モデル・金額計算・運営の照合/送金オペ・問題報告連動・未払い対応・通知・遷移**を補う。

精算フローは**3者横断**（レストラン＝買い手／農家＝売り手／運営＝管理画面）。配送完了後、**お金が必ずFishLinkを通って農家に届くまで**を扱う＝**中抜き対策の本丸（決済中継）**。関連：配送完了の起点は `fishlink-chat-spec.md`（配送管理チャット）／農家側注文は `fishlink-farmer-order-spec.md`。

### 実装方針の優先順位（他spec共通）
- 状態・金額・照合/送金オペ・遷移 → 本書とモックの通り。
- 見た目（色・タイポ）→ 既存DS準拠。**金額＝青 `#0B5FB0`／キャンペーン割引＝amber／緑はGAqP・成功・送金説明専用／赤はdanger専用。CTA＝青。**

---

## 1. 概要

- 受け取り時にレストランが **KHQR（ABA PAY）で FishLink 口座へ振込** → 運営が入金を照合 → 運営が農家へ送金 → 取引完了。
- **お金は農家に直接渡らず必ず FishLink を経由**（決済中継＝直接取引／中抜きを決済レベルで防ぐ最強の手）。
- 現状の決済基盤は**個人の銀行口座QR（ABA）**＝動的QR・自動消込は不可 → **運営が手動で照合・送金**（MVP）。将来はマーチャント決済へ昇格（§14）。

## 3. 状態モデル（5ステップ）

**運営の状態チップが正本**。各アプリは自分に関係する分だけ・**立場で呼んでよい**（全画面で1語に統一しない）。揃えるのは①順番②同じ瞬間に矛盾表示を出さないこと。

| 運営（正本） | レストラン | 農家 | 意味 |
|---|---|---|---|
| 受け取り完了 | お支払い待ち | 受取完了 | 配送完了・魚が届いた／支払い前 |
| （支払い完了 押下後） | 支払い確認待ち | 支払い確認待ち | レストランが「支払い完了」を報告・運営の入金確認待ち |
| レストラン入金済 | 支払い済み | （送金準備中） | 運営が入金を確認した |
| 農家送金済 / 取引完了 | 完了 | 送金済み・完了 | 運営が農家へ送金・取引終了 |

- 状態は**単一の真実**を3アプリで共有し、表示文言だけ役割別。

## 4. 金額モデル

- **手数料＝両側課金**（運営が料金設定画面で料率を自由設定／固定上限なし）。
  - **レストラン側＝支払額に上乗せ**（サービス料率・デフォルト5%）。買い手向け画面では**価格に内包**して表示（手数料項目は出さない）。
  - **農家側＝受取額から控除**（サービス料率・デフォルト5%）。
  - **キャンペーン**はレストラン側・農家側それぞれ独立に有効化＋割引率（デフォルト各2.5%＝半額）＋終了日。
- **送料は農家へ全額パススルー**（手数料を取らない）。
- 例（魚代37,500・送料12,400・各5%・キャンペーン各2.5%／FL-260604-003）：
  - レストラン支払 ＝ 39,375（魚代＋5%内包）＋送料12,400 −キャンペーン割引938 ＝ **50,837**
  - 農家受取 ＝ 魚代37,500 −手数料1,875 ＋キャンペーン938 ＋送料12,400 ＝ **48,963**
  - 運営の取り分 ＝ 50,837 − 48,963 ＝ 1,874
- **農家向けの手数料は分解表示**：「手数料（5%）-1,875 ＋ キャンペーン割引 +938」（圧縮した -937 にしない＝承認画面と統一）。

## 6. レストランの支払い（①②）

- **KHQR（ABA PAY）で FishLink 口座へ振込**。QRは個人口座QR＝**金額はレストランが手入力** → **「金額をコピー」**で入力ミスを低減。スキャン不可時は「リンクを開く／QRを保存」。
- 手数料は価格に内包（買い手は手数料項目を見ない）。明細に出すのは**キャンペーン割引のみ**（amber）。合計＝青。
- フッターは **「支払い完了」青のみ**（＝振込後に押す報告ボタン）。押下→**自前モーダル**「支払いを完了しましたか？／運営が入金を確認します。確認後に『支払い確認待ち』になります。」（キャンセル／はい）。ブラウザ標準ダイアログは使わない。
- **レビューは支払い前は出さない**。「支払い完了」押下後（支払い確認待ち）に「レビューを送信」が表示される（レビュー解禁＝支払いの合図の後）。

## 7. 運営の照合・送金（⑤⑥）＝運営照合型（MVP）

- **支払い確認方式＝運営照合型**：レストランの「支払い完了」は**報告**であり、それだけでは送金しない。**運営が入金を確認してから送金**（虚偽・誤タップは運営確認で吸収）。
- **⑥注文管理詳細**：レストラン入金確認 → 農家送金 → 取引完了 の**順次手動**（前段が済むまで次は非活性）。
  - **入金確認**：ABAアプリ（Merchant App）で入金を照合（振込元名義＋金額＋時刻）→「入金確認済にする」。**入金控え（入金スクショを添付）＋メモ**を毎回残す（証拠・過不足/振込元の補足）。
  - **農家送金**：運営が手動で農家のABA QRへ振込。**「送金額をコピー」**（正確な額入力）＋**「送金スクショを添付」**（送金控え＝証拠）。送金先名義／QR／リンクを表示。
  - **取引完了**：入金確認＋送金が済んだら「取引完了にする」。
- **⑤取引一覧**：KPI（取引数／流通総額／手数料）＋状態フィルタ。**「未払い（期限超過）」フィルタ＋件数バッジを追加**（期限超過を運営が見つけて督促）。
- 内部メモ（取引単位）は別途あり。色は内部用のため最小限（金額は濃色・割引amber）。
- **自動確定は作らない**：取引完了は運営手動。送金は当日バッチ運用で問題報告の猶予が自然に入る（§8）。滞留防止は運用目安のみ。

## 8. 問題報告との連動／送金タイミング（③④）

- **送金タイミング＝当日（夕方／営業後にまとめ送金）**。「朝受取 → その日の夕方に送金」の**数時間が問題報告の猶予**（鮮魚は受取時に良し悪しが分かるので数時間で十分）。**問題報告された注文だけ**バッチから外して保留・他は当日どおり送金。即時送金はしない。
- **問題報告**（窓口＝注文カード＋支払い画面・**チャットには置かない**）→ 運営がその注文の**農家送金を保留** → 調査 → **「送金 / 返金 / 一部返金」**：
  - **送金**＝問題なし/農家に非なし → 予定どおり農家へ送金。
  - **返金**＝問題あり正当（不着・傷み等）→ レストランへ返金・農家には送金しない。
  - **一部返金**＝部分的問題（5kg中2kg傷み等）→ 一部レストラン返金／残り農家送金。
  - 返金は**MVPは運営手動**（FishLink口座→レストラン）。
- **農家の表示（④）**：送金保留中は「**確認中（問題が報告されています）**」＋受取予定額（保留中）。何も出さないと「なぜ入金されない？」となるため理由を明示。
- **将来オプション**：即時送金＋問題時は農家の**次回送金で相殺**（リピート取引を利用）。

## 9. 未払い時（期限超過）

- **魚は渡済み＝キャンセル不能・与信（回収）の問題**（自動キャンセル／自動辞退はしない）。
- MVP＝**督促push＋運営フラグ・自動処分なし**。期限（例 08:48）はソフト目安＝督促トリガー。
  - 超過 → レストランへ**督促push**（実装済：配送完了時「○:○○までにお支払い」＋超過時「支払期限が過ぎました」）＋運営の取引一覧に「未払い（期限超過）」で上がる → 運営が対応。
  - 繰り返し督促の頻度は運営運用（自動連投はしない）。
- **常習・悪質は運営がアカウント制限**（アカバン機能あり）。ルール厳格化は後フェーズ。
- 農家保護＝FishLink経由で運営が回収を追える。将来は立替／保証も検討。

## 10. 通知

- 配送完了 → レストランへ「お支払いのお願い」push（期限つき・実装済）。
- 支払い期限超過 → 督促push（実装済）。
- 運営の入金確認・送金・取引完了 → 必要に応じて両者へ進捗通知（メッセージ/ステータス）。
- 集約はしない（都度）。

## 11. 表示ルール（色）

- **金額（合計・受取予定額）＝青 `#0B5FB0`**。キャンペーン割引＝**amber**。手数料・送料＝**中立グレー**（±符号で方向）。
- CTA（お支払いへ進む／支払い完了）＝**青**。緑はGAqP・成功・送金説明（③の green box）専用。赤は danger 専用（チャット内では使わない）。
- ④確認中の説明ボックスは**中立グレー**（緑にしない＝問題保留の状態に色を合わせる）。
- 全画面に**FL番号**を表示（問い合わせ・照合・サポート参照用）。
- 運営画面は内部用のため色は最小限でよい。
