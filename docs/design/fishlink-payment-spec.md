## 1. 概要・スコープ

**1.1 位置づけ** — 本specは案B（前払い＋ウォレット）のお金の流れのSoT。

**1.2 案Bとは**
- レストラン（買い手）は**注文時にKHQRで前払い**。FISHLINKが受取まで預かる（**エスクロー**）。
- 農家が**承認→配送→受取確認**で、FISHLINKから農家へ送金（手数料を引いた額・**配送完了後**）。
- 農家の**辞退／受取の問題／承認前キャンセル**時は、買い手へ**即ウォレット返金**。
- ウォレット残高は**次回注文に自動充当**。出金は**運営チャット申請**（全額・返金先へ）。

**1.3 対象範囲**
- 買い手：支払い画面（前払い）／ウォレット／注文状況
- 農家：注文確認（前払い済み表示）／配送管理／売上・取引
- 運営：入金照合・送金・返金・問題対応

**1.4 旧（受取時払い）からの主な変更**
- 支払いタイミング：受取時 → **注文時（前払い）**
- **「お支払い待ち」注文を廃止**（払って初めて成立）
- 辞退／問題／キャンセル：銀行返金 → **即ウォレット返金**
- **ウォレット新設**（買い手専用）
- クーポン：農家割り当て → **カート全体に1回**
- 農家の受取額・送金タイミング（配送完了後）は**不変**

## 2. お金の流れ（全体図）

**2.1 正常系**
1. **注文・前払い**：カート確定 → KHQRで前払い（ウォレット残高があれば自動充当＋残額KHQR）
2. **預かり（エスクロー）**：FISHLINKが入金を受け取り、運営が入金照合（状態＝支払い確認中）
3. **農家へ**：注文が農家に届く（〔支払いました〕で即農家へ・運営が並行で確認）→ 農家が承認
4. **配送**：準備 → 配送 → 配送完了
5. **受取確認**：問題報告がなければ配送完了から数時間後に自動で取引完了
6. **農家送金**：取引完了後、手数料を引いた額を農家へ送金（送料はパススルー）
7. **完了**：レビュー解禁

**2.2 返金が起きる分岐**
- 農家が辞退（手動／承認期限切れの自動辞退）→ その農家の小計を即ウォレット返金
- 承認前に買い手がキャンセル → その注文を即ウォレット返金
- 受取で問題報告 → 運営が確認 → 返金 or 送金保留（確認中）
- 入金が確認できない（未確認）→ 買い手がもう一度支払う or 運営が照合

**2.3 ウォレットの循環**
返金はウォレット残高に入る → 次回注文で自動充当 → 残額だけKHQR。出金は運営チャットに申請（全額・返金先へ）。

## 3. 状態モデル（状態機械）

**3.0 単位＝農家ごとの注文** — 複数農家カートでも注文は農家ごとに独立。状態・支払い・返金は農家ごとに進む（前払いだけまとめて1回KHQR）。

**3.1 メインの流れ（正常系）**
```
支払い確認中 → 承認待ち → 準備中 → 配送中 → 配送完了 → 取引完了 → 送金待ち → 送金済
```

**3.2 同じ注文の「買い手の顔」と「農家の顔」**

| 注文の状態 | 買い手（注文状況） | 農家（注文確認） |
|---|---|---|
| 支払い確認中 | お支払い確認中 | 承認待ち（前払い済み） |
| 承認待ち（入金確認済） | 農家確認中 | 承認待ち |
| 準備中／配送中 | 進行中 | 準備中／配送中 |
| 配送完了 | お届け済み | 配送完了（送金待ち） |
| 取引完了→送金 | 完了 | 送金待ち → 送金済 |

**3.3 分岐**
```
支払い確認中 ──(入金確認できない)──→ 支払い未確認 →〔もう一度支払う/運営照合〕
承認待ち ──(辞退/承認期限切れ自動辞退)──→ 辞退 → 即ウォレット返金
承認待ち ──(買い手が承認前にキャンセル)──→ キャンセル → 即ウォレット返金
配送完了 ──(問題報告)──→ 確認中(送金保留) → 運営確認 → 返金 or 送金続行
```

**3.4 終端状態**
- 送金済（正常完了）／返金済（辞退・キャンセル・問題）
- ※カートの全農家が辞退した場合はクーポンも未使用で戻る

## 4. お金のルール（横断）

### 4.1 前払い・エスクロー・即農家へ・運営照合
- **前払い**：カート確定時にKHQRで前払い（残高があれば自動充当＋残額KHQR）。注文は**支払い完了で初めて成立**（未払い注文は存在しない）。在庫は支払い完了で引く。
- **エスクロー**：前払い金はFISHLINKが受取まで預かる。支払い画面に安心文「🔒 受け取りに問題があれば返金されます」。
- **即農家へ＋運営照合**：〔支払いました〕で即農家へ（支払い確認中／農家は承認待ち）。運営が並行で入金照合 → 確認できたら確認済（買い手：農家確認中へ）。将来Bakong連携で自動確認＝ほぼ即時。
- **支払い未確認（例外）**：運営が入金を確認できない → 買い手に「お支払い未確認」（〔もう一度支払う〕／〔運営に問い合わせる〕）。**自動キャンセルしない**（農家準備中の恐れ＋払ったのに照合できてない場合は金が宙に浮く）→ 運営が照合／督促で解決。副作用＝最悪「魚は行ったのに未確認」が手動照合失敗時のみ再発（→§8運営対応）。

### 4.2 手数料・送料・キャンペーン（農家の受取額）
- **手数料**：対象＝**魚代のみ**（送料にはかけない）。率＝**変動**（標準5%・運営パラメータ）。立ち上げ期はキャンペーンで実質ゼロ/低く（反現金の武器）。
- **送料**：レストラン負担。**農家へ全額パススルー**（手数料対象外）。農家が配達を手配・負担するため受取額に乗る。
- **キャンペーン**：「通常の手数料を引いてキャンペーンで戻す」形。農家にはプラス表示（amber）。買い手側にも割引が反映。**クーポンと併用不可**（お得な方を自動適用）。
- **農家の受取予定額（例）**：`魚代 − 手数料 + キャンペーン戻し + 送料`＝ 37,500 − 1,875 + 938 + 12,400 = **48,963 KHR**。

### 4.3 クーポン（カート全体に1回＋複数農家の返金）
- **適用**：クーポンは**カート全体に1回の割引**（農家に割り当てない）。期限が近い順に自動適用（コード手入力・手動選択なし）。1注文1枚。キャンペーンと併用不可。
- **複数農家の返金**：1農家辞退 → その農家の**小計を満額ウォレット返金**・割引は**残った農家に残る**。全農家辞退 → **全額返金＋クーポン未使用で戻る**。按分・農家割り当てしない（返金額が一意）。
- **例**：大星51,775＋パイリン32,000＝83,775、クーポン−2,500→払う額81,275。大星辞退→51,775返金（割引−2,500はパイリンに残る）。

### 4.4 ウォレット充当（支払い時）
- **既定＝全額自動充当**＋〔変更〕で金額指定（0＝使わない）。クーポンと同流儀（既定で使う・外せる）。
- **順番**：合計（クーポン適用後）→ ウォレット充当 → 残額KHQR。残高≥合計ならKHQR不要で〔注文を確定する〕。
- ウォレットは**買い手専用**（農家は送金で受取＝ウォレット不要）。

### 4.5 返金（辞退／問題／キャンセル → 即ウォレット）
- **ケース**：辞退（手動／自動）＝その農家の小計を満額／承認前キャンセル＝その注文を満額／問題報告＝運営確認後に返金（全額/一部）or 送金続行。
- **返金先＝ウォレット（即時）**：銀行返金ではなく残高に即クレジット（待ち時間ゼロ＝案Bの核）。買い手に明記「✓ ◯◯KHRをウォレットに返金しました（次回使えます）」＋プッシュ。
- **返金額**：その農家の小計を満額（魚代＋送料・内臓処理込み）。支払い方法に関わらず全額がウォレットに入る（ウォレット分もKHQR分もまとめて残高に戻る）。クーポン絡みは§4.3。

### 4.6 農家送金のタイミング（配送完了後・問題報告で保留）
- **解禁条件**：配送完了 → 問題報告ウィンドウ（数時間／配送完了＋4h目安）→ 報告なしで自動確定（取引完了）→ 送金解禁。報告ありは「確認中」で保留。
- **実行**：取引完了後、送金待ち → 運営が手動送金（当日夕方バッチ＋数時間窓）→ 送金済。自動化（マーチャント決済昇格）は将来。
- **問題報告時の保留**：「確認中」で送金保留 → 運営確認 → 返金（買い手ウォレット）or 送金続行（一部返金あり）。農家の売上・取引で「確認中（問題報告あり・保留）」表示＝サマリで「本日送金予定」と別枠。
- **不変点**：早まったのは「買い手が払う瞬間」だけ。農家の受取額・送金ロジック・タイミングは不変。

### 4.7 出金（ウォレット残高の返金申請）
- **方式＝運営チャット申請（自動出金なし）**：〔残高の返金を申請〕→ 確認シート →〔運営に申請する〕→ 運営チャット → 運営が返金先へ手動送金 → 残高減算。
- **金額＝全額のみ（MVP）**：残高ぜんぶ。部分指定は作らない。申請すると残高はすぐ差し引かれる（1〜3営業日で振込）。
- **返金先**：買い手の返金先（口座／QR）。農家の「送金先」とは別物。ウォレット申請シートと買い手マイページの返金先は**同じ1件を一元管理**。未登録なら申請時に登録を促す（返金先JIT）・申請ボタンは登録まで無効。
- **見せ方**：残高を主役・出金は控えめなテキストリンク。残高0では出金リンク非表示。

## 5. データモデル

**5.1 cart（前払いの単位＝複数農家をまとめる）**
```
cart {
  cartId, buyerId,
  orders: [orderId...],        // 農家ごとのsettlement
  couponId, couponDiscount,    // カート全体に1回（§4.3・農家に割り当てない）
  walletAppliedTotal,          // ウォレットから充当した合計
  khqrPaidTotal,               // KHQRで払った合計
  paidAt,                      // 前払い完了時刻（=注文成立）
}
```

**5.2 settlement（農家ごとの注文）**
```
settlement {
  orderId, cartId, buyerId, farmerId,
  status,                      // §3の状態
  fishPrice,                   // 魚代（買い手価格・手数料内包）
  deliveryFee,                 // 送料（レストラン負担・農家パススルー）
  campaignDiscount,            // キャンペーン割引（amber）
  subtotal,                    // この農家の小計（=辞退時の返金額）
  serviceFeeRate,              // 手数料率（変動・標準0.05・魚代のみ）
  farmerPayout,                // 受取予定額＝魚代−手数料+キャンペーン戻し+送料
  paidReportedAt, depositConfirmedAt, depositMemo, depositProofUrl, payerName,  // 入金照合（運営）
  remittedAt, remitProofUrl,                                                    // 送金（運営）
  problemReportId, resolution, refundAmount, refundToWalletAt,                  // 問題・返金（ウォレットへ）
  approvalDeadline,            // 承認期限（切れたら自動辞退）
}
```
※クーポンは**cart単位**（settlementに割り当てない）。辞退の返金＝その農家の`subtotal`を満額。全農家辞退でcouponは未使用に戻す（§4.3）。

**5.3 wallet（買い手専用）＋ transaction（履歴）**
```
wallet { buyerId, balance }

transaction {                  // ウォレット履歴の1行
  txId, buyerId, amount,       // +増 / −減
  type,                        // 辞退の返金 | 受け取りの問題の返金 | 注文に使用 | 残高の返金（出金）
  counterpart,                 // 農家名 or 注文FL番号
  relatedOrderId, createdAt,
}
```

**5.4 refundDestination（買い手の返金先）**
```
refundDestination {            // ＝出金の振込先。農家の「送金先(payout)」とは別物
  buyerId, method,             // ABA口座 | QR
  accountName, qrUrl, qrImageUrl,   // マイページと一元管理・未登録なら出金申請時にJIT登録
}
```

## 6. ウォレット詳細（残高・履歴4種・出金申請）

mock: `fishlink-wallet-mockup.html`（4状態）。入口＝買い手マイページ「💰ウォレット 残高◯◯」行（返金先の上・お金グループ）。買い手専用。

**6.1 画面構成** — 残高カード（大きく）→〔残高の返金を申請〕（控えめリンク）→ 利用履歴。

**6.2 残高・ナッジ** — 残高を大きく青。ナッジ「💡 次回の注文で自動的に使われます」＋サブ「支払い時に使う金額を選べます」。見せ方＝残高主役・出金控えめ。残高0では出金リンク非表示。

**6.3 利用履歴（4イベント種別）** — チャージ機能は無い（入金は返金のみ）。

| type | 例 | 符号・色 |
|---|---|---|
| 辞退の返金 | +51,775（大星農場・FL…） | ＋緑 |
| 受け取りの問題の返金 | +6,825（パイリン農場・FL…） | ＋緑 |
| 注文に使用 | −23,500（FL…） | −黒 |
| 残高の返金（出金） | −11,600 | −黒 |

各行＝日付・理由ラベル・相手（農家名 or FL番号）・金額。

**6.4 出金（残高の返金を申請）フロー**
1. 〔残高の返金を申請〕（残高>0の時だけ）→ 確認シート
2. 確認シート＝返金額（残高ぜんぶ）／返金先（ABA・名義〔変更〕）／🔒「運営が手動振込・1〜3営業日・申請で即減算」
3. 返金先未登録 → amber「返金先が登録されていません」＋〔返金先を登録する〕（JIT）・〔運営に申請する〕は無効
4. **6.4.5 〔運営に申請する〕押下後**：①残高を**即減算**＋履歴「残高の返金（申請）−◯◯」を記録（二重使用防止）②トースト「返金を申請しました」→**運営チャットへ直行**し**申請カード**を投稿（残高の返金 ◯◯KHR／返金先）＋運営案内「1〜3営業日で返金」（完了画面は挟まない）③ウォレットに「返金を申請中：◯◯KHR」バナー（処理中）④運営が送金完了→チャット通知＋履歴確定。mock＝`fishlink-support-chat-mockup.html` ③'（運営チャットに申請カード）。

**6.5 空状態（新規・残高0）** — 残高0（グレー）＋説明「農家の辞退・受け取りの問題の返金がここに貯まり、次回の注文で自動的に使われます」。出金リンク非表示・履歴は空。

## 7. 通知

**7.1 買い手への通知**
- 農家が承認 → 「農家が注文を承認しました」
- **辞退（手動／承認期限切れの自動辞退）** → 「◯◯が辞退しました。◯◯KHRをウォレットに返金しました（次回使えます）」
- **受け取りの問題の返金** → 「◯◯KHRをウォレットに返金しました」
- お支払い未確認 → 「入金を確認できませんでした。もう一度お支払いください」
- 出金：申請受付（運営チャット内）／**返金完了** → 「返金しました」
- 配送状況（配送中・配送完了）→ 配送管理チャットの系統メッセージ

**7.2 農家への通知**
- **新しい注文（前払い済み・承認期限◯）** ＋期限10分前リマインド（既存）
- 問題が報告された → 「問題が報告されています（送金確認中）」
- **送金済** → 「**◯◯KHRを送金しました。取引相手を評価しましょう**」＝**送金通知でレビューを誘導**（レビュー解禁は送金済＝案1・"お金が入った"瞬間がレビューの最良フック）。タップでレビュー画面へ。未記入なら**翌日リマインド1回**（しつこくしない）。相互レビュー（レストランも評価）で率を底上げ。

**7.3 廃止された通知（旧・受取時払い）**
- ❌「お支払いのお願い」push（配送完了時）＝廃止（前払い済み）
- ❌ 支払い期限超過の督促push＝廃止（未払い注文が存在しない）

**7.4 方針** — 集約しない（都度）。特に**「お金が動いた」通知（返金・送金）は確実に出す**＝案Bの安心の核。

## 8. 運営オペレーション（入金照合・送金・返金・問題対応・未確認）

> ⚠️ 本§は**お金の流れのオペ・ロジック（SoT）**。実際の**運営管理画面のUI/レイアウト（取引一覧・注文管理詳細）は別途「運営管理画面レビュー」で詰める**。既存の運営mock（`fishlink-payment-mockup.html` ⑤⑥）は**受取時払いベース＝案B用に更新が必要**（前払い照合・ウォレット返金・確認中・出金申請対応・「未払い→お支払い未確認」）。

**8.1 役割（案B）** — 入金照合（前払い）・農家送金（配送完了後）・返金（辞退/問題）・出金（ウォレット返金）・問題対応・未確認の解決。MVP＝運営照合型・手動（個人口座QR）。将来マーチャント決済で自動化（§9）。

**8.2 入金照合（前払い）** — 買い手の〔支払いました〕は報告。運営がABA Merchant Appで照合（振込元名義＋金額＋時刻）→ 確認済に。**入金控え（スクショ）＋メモ**を残す。確認できない → お支払い未確認（自動キャンセルせず照合/督促・§4.1）。

**8.3 農家送金（配送完了後）** — 取引完了（配送完了＋問題報告窓経過）→ 送金待ち → 運営が手動送金（当日夕方バッチ＋数時間窓）。**送金額コピー＋送金スクショ添付**。送金先＝農家プロフィール登録（未登録は送金不可）。問題報告あり → 確認中で保留。

**8.4 返金（ウォレットへ）** — 辞退／承認前キャンセルは自動でウォレット返金。問題報告は運営が**送金／返金／一部返金**を判断 → 返金分は買い手ウォレットへ。

**8.5 出金（残高の返金申請）** — 運営チャットの申請カード → 運営が買い手の**返金先（口座/QR）へ手動送金**（1〜3営業日・送金スクショ）。残高は申請時に即減算済み。返金先未登録なら申請時にJIT登録。

**8.6 取引一覧（運営画面）** — 状態フィルタ。旧「未払い（期限超過）」→ **「お支払い未確認（照合できない例外）」に置換**。内部メモ・色は最小限。詳細UIは運営管理画面レビューで。

**8.7 自動化（将来）** — ABA PayWayマーチャント／Bakong：動的QR＋入金webhook（自動照合）＋送金API。入金の自動照合で「支払い確認中」がほぼ即時・未確認の例外も激減。

## 9. エッジ・未確定

**9.1 確定済みのエッジ（本specで処理済み）**
- 複数農家カートの部分辞退（農家ごと独立・§3.0／§4.3）
- 承認期限切れ → 自動辞退（§3.3）／承認前キャンセル（§4.5）
- お支払い未確認（§4.1）／問題報告 → 送金保留（§4.6）
- 承認後に農家がキャンセルしたい時 → **運営チャット経由**（専用ボタンなし）

**9.2 要確認（設計でなく外部・案B詰めE/F）**
- **E 預かり金（前払い金・ウォレット残高）の規制・信用面** ＝ カンボジアの電子マネー／決済ライセンス等 → 法務・運営で要確認
- **F レストランが前払いを嫌がるリスク** ＝ 設計で緩和済（即ウォレット返金・出金申請・エスクロー表示）だが実店舗ヒアリングで検証

**9.3 次フェーズ**
- 🔴 **決済自動化（必須・fast-follow）** ＝ ABA PayWayマーチャント／Bakong（下道さん＋代表でABA可否確認）
- 部分出金（今は全額のみ）／掛け払い（後払い）／即時送金＋問題時は次回送金で相殺
- レストラン向け領収書・取引履歴（B2B経理）／端数処理（0.5 KHR丸め）

**9.4 別レビューで詰める**
- **運営管理画面**（取引一覧・注文管理詳細）の案B更新
- 農家の**送金先**登録（プロフィール画面）／買い手の**返金先**登録（マイページ）

## 11. 表示ルール（色・UI規約）

- **金額（合計・受取予定額・お支払い額）＝青 `#0B5FB0`**。割引（キャンペーン・クーポン）＝**amber**。手数料・送料＝**中立グレー**（±符号で方向）。
- **CTA＝青**。緑はGAqP・成功・**返金済の安心**（辞退カードの「✓ 返金しました」）専用。赤はdanger専用。**全画面にFL番号**。運営画面は内部用で色は最小限。
- レビューは**完了後のみ**解禁・レビュー送信は**副CTA（白地・青枠）**。
- 確認系モーダルは**中央モーダル**で統一（ボトムシートは入力・選択用）。
- 問題報告の窓＝**配送完了＋4h**（`fishlink-trouble-report-spec.md` §4・N=4h）。窓内だけ〔問題を報告〕表示・窓後は運営チャットへ。農家の問題報告＝自分の送金は止めず運営にエスカレ（受取対応/やり取り/その他）。買い手＝商品の問題6カテゴリ → 運営が農家送金を保留 → 返金判断。
- 「送金について」情報ノートは**全状態で中立グレー**（状態色は「現在の状態」ボックスが担う）。
