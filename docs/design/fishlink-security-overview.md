# FISHLINK セキュリティ対策 概要

作成日: 2026-07-02 ／ 対象: FISHLINK（カンボジア向け B2B 水産マーケットプレイス・PWA）
基盤: Firebase（認証・データベース Firestore・サーバー処理 Cloud Functions・ファイル保管 Cloud Storage・プッシュ通知 FCM）

## 概要

FISHLINK は「お金・個人情報・取引データ」を扱うアプリのため、**サーバー側で強制する仕組み**を中心に安全対策を組んでいます。要点は次の3つです。

1. **データは持ち主と運営しか触れない** — 誰がどのデータを読み書きできるかを、アプリ（クライアント）任せにせず、サーバー側のルールで強制しています。
2. **お金（ウォレット残高）はアプリから書き換え不可** — 残高の増減はサーバー処理（Cloud Functions）だけが行い、利用者の端末からは残高を直接変更できません。二重支払い・二重返金も防止しています。
3. **中抜き（アプリ外での直接取引）を防止** — 連絡先（電話番号・URL・SNS）が本文に紛れ込んでも、サーバー側で自動的に伏字にします。手元のアプリを改造しても回避できません。

以下、① 現在の対策 と ② 追加候補（コスト別）を整理します。実コードに基づいて記載しています。

---

## ① 現状の対策

### 認証（ログイン・本人確認）

- **電話番号でのログイン**: 利用者は電話番号だけを入力します。内部的には電話番号を正規化し、`電話番号@fishlink.local` という形式に変換して Firebase の認証に載せています（利用者は意識しません）。表記ゆれ（`012 345 678` / `+855…` など）は1つの形式に統一し、カンボジアの番号形式かどうかも検証します。
- **総当たり攻撃に強いエラー表示**: ログイン失敗時は「電話番号またはパスワードが違います」とだけ返し、**どちらが間違っているか・その番号が登録済みかを伝えません**。攻撃者が有効な番号を探り当てる手がかりを与えない設計です。
- **SMS ワンタイムコード（OTP）による本人確認**: パスワード再設定・電話番号変更では、Firebase の電話認証（SMS-OTP）で番号の所有を確認します。**サーバー側でこの確認結果（IDトークン）を再検証してから**、管理者権限でパスワードやメール（＝ログインID）を更新するため、なりすましで他人のアカウントを乗っ取ることはできません。
- **不正利用者の強制ログアウト**: 運営がアカウントを利用停止（BAN）にすると、次回のアクセス時・ログイン時に自動的にログアウトされ、利用を継続できません。
- **ログイン状態の維持**: 正規の利用者はログインしっぱなしにできますが、ログアウト時には端末に残ったキャッシュ・画像・通知トークンを消去し、共用端末での情報残留を防ぎます。

### アクセス制御（誰が何を読み書きできるか）

サーバー側のルールで、コレクション（データの種類）ごとに権限を厳格に定めています。アプリ側のコードに関係なく、この境界は破れません。

- **注文**: その注文の当事者（買い手の飲食店・売り手の農家）と運営だけが読み書きできます。第三者は他人の注文を見られません。
- **プロフィール（users）**: 本人だけが自分の情報を更新できますが、**利用停止フラグや GAqP 認証フラグなど「重要な項目」は本人が触れません**（認証フラグは運営のみが更新可能）。
- **クーポン**: 発行はサーバー処理のみ、利用者は「使用済みにする」以外の変更ができず、未使用のときだけ適用でき、削除は不可。金額を書き換えたり使い回したりできません。
- **問題報告・レビュー**: 報告は本人と運営のみ閲覧、レビューの非表示化（モデレーション）は運営のみが実行できます。
- **運営チャット**: 利用者は「自分の発言」しか投稿できず、**運営や自動メッセージへのなりすまし投稿を封じています**。対応状況（未対応/対応済み）の項目は運営だけが書き換えられ、利用者が自分で「対応済み」に偽装して問い合わせを隠すことはできません。
- **各種カウンター・お知らせ・やることリスト**: 利用者からの直接書き込みを禁止し、サーバー処理経由のみに限定しています。

### お金の保護（ウォレット・精算）

- **残高はアプリから書き換え不可**: ウォレットの残高と入出金履歴（台帳）は、**利用者の端末からは一切書き込めない設定**にしています。残高の増減（注文時の充当・辞退時の返金・出金申請）は、すべてサーバー処理（Cloud Functions）だけが行います。これがお金の改竄を防ぐ中核です。
- **二重支払い・二重返金の防止**: 通知の再送などでサーバー処理が二重に走っても、注文ごとに「処理済みの印（マーカー）」を付けて1回しか実行しないようにしています（支払い充当・返金・取引件数の集計すべてに適用）。
- **返金先が未登録なら出金申請不可**: 出金（残高の返金申請）は、返金先の登録がないとサーバー側で拒否されます。誤送金・不正申請の入口を塞いでいます。
- **前払い・エスクロー方式**: 代金は注文時に前払いされ、運営（FISHLINK）が一旦預かってから農家へ渡す流れです。お金が必ず FISHLINK を通ることで、取引の追跡と中抜き防止を両立します。

### データ・ファイルの保護（Cloud Storage）

- **ファイルはパスごとに権限を限定**（許可リスト方式）: 商品写真・プロフィール画像は公開読み取り可、送金先QR・精算スクリーンショット・問題報告の画像などは本人または運営のみ、というように用途ごとにアクセスを分けています。
- **音声メッセージ**: その注文の当事者だけが読み書きでき、さらに**5MB 未満・音声形式のみ**という制限を付けて、不正なファイルのアップロードを防いでいます。
- **精算スクリーンショット**: 入金控え・送金控えは運営のみが閲覧でき、一般利用者には非公開です。

### 不正・スパム・中抜き対策

- **連絡先のサーバー側自動マスク**: レビュー・公開質問（Q&A）・運営チャットで、電話番号・URL・SNS のIDなどが本文に含まれていると、**サーバー側で自動的に伏字**にします。アプリを改造して送っても、サーバーが再度チェックするため回避できません（アプリ外での直接取引＝中抜きの防止）。
- **管理者専用処理の権限確認**: 利用停止・OCR・価格取り込み・各種データ移行などの管理機能は、呼び出すたびに**その利用者が本当に運営（admin）かをサーバー側で確認**します。運営でなければ拒否されます。また、運営でも「自分自身や他の運営を利用停止にする」ことはできません。
- **SMS 悪用対策**: SMS-OTP はカンボジア（+855）宛てに限定し、送信前に reCAPTCHA（人間確認）を通すことで、SMS の大量送信によるコスト攻撃を抑えています。

### 通信

- **常時 HTTPS**: すべての通信は暗号化されています（Firebase Hosting）。
- **通知**: プッシュ通知は Firebase の仕組みを利用し、遷移先リンク付きで安全に配信します。

## ② 追加候補（コスト別）

現状でも実運用に足る対策を備えていますが、規模拡大・投資家評価・監査対応を見据え、以下を候補として整理します。コスト感は「無料〜低額（設定中心）／中（実装工数あり）／高（外部サービス・継続費用）」で示します。

| 対策候補 | コスト感 | 目的（なぜ必要か） |
|---|---|---|
| **App Check の適用** | 無料〜低額（設定＋一部実装） | 現状、App Check（正規アプリからのアクセスかを検証する仕組み）は**未導入**。導入すれば、正規アプリ以外からのデータベース・サーバー処理への直接アクセスを大幅に減らせる。優先度の高い候補。 |
| **注文ステータスのサーバー側強制** | 中（実装工数あり） | 現状、注文の承認・辞退・キャンセルはアプリ側から直接更新する信頼モデル。これをサーバー処理（callable）経由に寄せ、状態遷移とフィールドを厳密に検証すれば、不正な状態変更をより強固に防げる。 |
| **レート制限（回数制限）** | 中（実装工数あり） | メッセージ送信・出金申請・ログイン試行などに回数制限がない。短時間の大量リクエスト（スパム・総当たり）を抑えるため、上限を設ける。 |
| **監査ログ** | 低〜中 | 利用停止・返金・送金などの管理操作の履歴を体系的に残し、後から「誰が・いつ・何をしたか」を追跡できるようにする。トラブル対応・不正調査・投資家/監査向けの説明責任に有効。 |
| **依存パッケージの脆弱性スキャン** | 無料（設定のみ） | npm audit / Dependabot を CI に組み込み、利用ライブラリの既知の脆弱性を自動検知。現状は自動スキャン未設定。設定のみで導入でき費用対効果が高い。 |
| **バックアップ・障害復旧** | 低額（設定＋保管費用） | Firestore の定期エクスポート（自動バックアップ）を設定し、データ消失・誤操作からの復旧手段を確保する。 |
| **WAF・DDoS 対策** | 高（外部サービス・継続費用） | Cloud Armor 等でアプリ層の攻撃・大規模アクセス攻撃を防御。取引量・注目度の増加に応じて検討。 |
| **決済自動照合の規制対応** | 高（外部連携・継続対応） | 将来 ABA / Bakong 等の自動入金照合を導入する際に必要となる、預かり金規制・決済関連のコンプライアンス対応。 |
| **重要操作のパスワード再認証（任意）** | 低〜中 | 電話番号変更・出金など影響の大きい操作で、パスワード再入力を求めるハードニング。現状はログイン済みセッションを認可境界としている。 |

## まとめ

FISHLINK は、**「データの持ち主と運営しか触れない」「お金はサーバーだけが動かす」「中抜きはサーバーで自動的に塞ぐ」**という3点を軸に、アプリ任せにしない（サーバー側で強制する）設計を採用しています。認証は電話番号ベース＋SMS 本人確認、権限はコレクションごとに厳格化、残高は書き換え不可、連絡先は自動マスク、と実運用に必要な基本対策は揃っています。

一方で、事業規模の拡大・投資家評価・監査対応を見据えると、**App Check の適用（優先）**、依存パッケージの自動スキャン、監査ログ、レート制限、バックアップ整備などは、比較的低コストで信頼性・堅牢性をさらに高められる有力な候補です。決済自動化・大規模防御（WAF/DDoS）は、取引量の増加に合わせて段階的に導入する方針とします。
