### 🔵 1-6.【設計相談＝要】ステータスを「配送軸」と「入金軸」の2つに分ける（前払い化の本質的な影響）
**背景**：現状ステータスは**1本の直線**（準備開始→配送開始→受け取り完了→レストラン入金済→農家送金済→取引完了）。これは**受取時払い時代の名残**で、順番が「受け取ってから入金」になっている。
- **前払いに変えたので**、買い手は最初に入金し、**注文は運営の入金確認を待たずに流れる**（＝入金確認は運営の並行作業・その日中に実施・注文をブロックしない）。
- ＝**1件の取引が「配送＝配送中／入金＝入金未確認」のように2つの状態を同時に持つ**。1本の直線ステータスでは表現できない（＝「配送中かつ入金未確認」が表せない）。

**提案＝ステータスを2軸に分ける**：
| 軸 | 状態の流れ | 例外 |
|---|---|---|
| **① 配送ステータス**（農家・買い手が進める） | 農家確認中 → 準備開始 → 配送開始 → 受け取り完了 → 取引完了 | 辞退・キャンセル |
| **② 入金ステータス**（運営が動かす） | 入金確認待ち → 入金確認済 → 農家送金済 | 未払い（期限超過）・返金済 |

**UI（取引一覧）への反映**：
- **ステータス列を2つに**（配送／入金のバッジ2個）。
- **フィルタも2軸**（配送で絞る＋入金で絞る）＝「配送中かつ入金未確認」も絞れる。
- **これが運営のTo-Doになる**：入金軸で「**入金確認待ち**」＝今日確認すべき入金の一覧／「**農家送金待ち**」（配送完了＋入金確認済）＝送金すべき一覧。前払いの運営作業（入金確認・農家送金）がそのまま絞り込める。

**この2軸化で他の指摘も自然に整理される**：
- 1-2（辞退・キャンセルのchip）→ 配送軸の例外として入る。
- 1-4（返金済タグ）→ 入金軸に「返金済」があるので、そちらで表現。
- 「レストラン入金済」の置き場所問題 → 入金軸に移って解消。

**要相談**：これは**データの持ち方（1フィールド→2フィールド）**の変更なので、下道さんに「①今ステータスは1本か ②配送軸/入金軸の2分割にできるか ③上記の状態マッピングで実装的に問題ないか」を確認したい。買い手の注文状況・農家画面のステータス表示にも波及する可能性があるので、影響範囲も含めて相談。※状態名・流れは叩き台（実装に合わせて調整可）。
**※正本に追記済＝`fishlink-payment-spec.md` §3.5「運営管理画面＝2軸ステータス（配送軸／入金軸）」**（1-6の設計はこちらがSoT）。

**モック＝`fishlink-admin-transaction-list-mockup.html`**（2軸版の取引一覧）。ステータス列2つ＋フィルタ2軸／「配送開始×入金確認待ち」など1本では表せなかった状態のサンプル／運営To-Do（入金確認待ち・農家送金待ち）をアンバーで／1-1キャンセルラベル・1-4返金済タグ・1-5 KPI注記も反映。
